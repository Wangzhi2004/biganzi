。

PRD v0.1：AI 长篇连载小说自治创作系统

产品暂定名：笔杆子·连载引擎

产品定位：面向长篇连载小说的 AI 创作操作系统。它不是一个“帮我写一章”的聊天工具，而是一个能持续管理百万字作品的创作系统，集写作、记忆、提示、维护、创意、审稿、排期于一体。

一句话定义：让 AI 像成熟长篇作者一样持续经营一部小说，而不是临时生成一段文字。

核心目标：帮助用户从一个故事种子开始，自动建立作品设定、规划长篇结构、生成章节、维护正史、管理伏笔、保持人物一致性、控制文风、评估追读，并支持长期自动连载。

核心判断：产品壁垒不在“生成正文”，而在 长篇记忆、正史维护、伏笔账本、人物弧线、风格一致性、节奏控制、自动审稿。

⸻

1. 背景与问题

当前 AI 写小说工具普遍存在几个问题：

第一，只会生成短文本，缺少长篇结构控制。写前 10 章还顺，写到 100 章后人物动机、伏笔、设定、战力、关系线开始漂移。

第二，缺少“正史”。大多数工具依赖上下文窗口或普通摘要，无法判断哪些内容是正式设定，哪些只是临时草稿，哪些是已经废弃的桥段。

第三，缺少章节责任。AI 常常把每章写成“看起来像小说”的文本，但不知道这一章究竟要推进主线、回收伏笔、强化人物，还是制造追读钩子。

第四，缺少维护能力。长篇连载最难的是持续维护。人物不能失踪，伏笔不能遗忘，设定不能冲突，节奏不能长期停滞。

第五，缺少风格稳定性。AI 写着写着容易滑向通用网文腔，作品辨识度下降。

第六，多 Agent 方案如果设计过重，会产生高延迟、高成本、高不可控。每章都让十几个 Agent 发言，工程上很臃肿，用户体验也会变慢。

所以本产品的设计原则是：

用系统工程替代 prompt 工程。用状态库替代长上下文幻想。用流水线替代多 Agent 混战。用可解释评分替代模型自嗨。

⸻

2. 目标用户

第一类：个人网文作者。
痛点是更新压力大、容易卡文、忘设定、后期崩盘、写作效率低。
核心需求是快速生成章节、管理设定、稳定更新、提高追读。

第二类：AI 写文创业者 / 内容工作室。
痛点是需要批量运营多本书，但人工编剧、编辑、审稿成本高。
核心需求是多项目管理、自动连载、风格复制、质量控制、生产流水线。

第三类：IP 策划团队。
痛点是世界观复杂、角色多、长线伏笔多，需要系统化管理。
核心需求是世界观数据库、人物关系图、剧情规划、跨媒介资产沉淀。

优先服务：个人作者 + 小型内容工作室。

⸻

3. 产品核心价值

产品要解决 6 件事：

一，帮用户从 0 到 1 创建一本可连载的小说。
包括类型、卖点、主角、世界观、主线、卷纲、前 30 章规划。

二，帮用户持续生成章节。
不是简单续写，而是按章节功能、场景卡、人物状态、伏笔账本生成。

三，帮用户记住一切。
人物、关系、事件、时间线、地点、能力、道具、组织、规则、伏笔、承诺全部结构化入库。

四，帮用户维护长篇。
自动发现设定冲突、人物漂移、伏笔遗忘、节奏停滞、重复桥段、水文风险。

五，帮用户形成风格。
提取样章风格，生成作品风格指纹，后续章节自动对齐。

六，帮用户自动连载。
生成章节队列，自动审稿，自动重写，自动排期，维持长期更新节奏。

⸻

4. 产品形态

产品不是聊天框，而是 小说 IDE。

主界面分为五个区域：

左侧：作品结构树
包含全书、卷、篇章单元、章节、场景。

中间：正文编辑器
支持写作、续写、改写、扩写、删减、增强冲突、增强钩子、调整风格。

右侧：叙事状态面板
展示当前人物状态、相关伏笔、世界规则、关系变化、风险提醒。

底部：时间线 / 事件流
显示每章发生了什么、造成了哪些状态变化、影响了哪些后续内容。

顶部：连载控制台
展示今日待写、待审、待发布、质量分、风险分、追读预测。

⸻

5. 核心设计原则

第一，少 Agent，多流水线。

不要设计十几个 Agent 长时间对话。产品内部采用：

1 个 Orchestrator 总控器
负责决定当前任务走哪条流程、调用哪些工具、如何组织上下文、如何写入数据库。

3 个核心智能角色
分别是 Planner、Writer、Auditor。

Planner：负责规划。
生成作品 DNA、卷纲、篇章单元、章节目标、场景卡。

Writer：负责写作。
根据规划、正史、风格、章节目标生成正文和改写正文。

Auditor：负责审稿。
检查一致性、节奏、风格、人物、伏笔、设定、追读。

其余能力全部工具化，不做独立 Agent。例如：

记忆召回工具。
伏笔查询工具。
人物状态查询工具。
风格检测工具。
章节评分工具。
冲突检测工具。
套路变体工具。
摘要入库工具。
时间线检查工具。

这样系统既有专业分工，又不会变成 Agent 会议室。

第二，正史优先。

模型生成的任何内容，在被确认前都只是草稿。只有写入 Canonical Story Graph 的内容才是正史。

第三，近景精确，远景模糊。

未来 300 章只保留方向，未来 30 章保留剧情单元，未来 5 章保留明确目标，下一章精确到场景卡。

第四，所有章节必须有功能。

每章生成前必须指定章节功能。不能出现“随便续写”。

第五，生成后必须入库。

每章生成完成后，系统自动抽取状态变化，并更新人物、关系、事件、伏笔、承诺、规则。

第六，质量不过线自动重写。

每章通过评分系统审查，低于阈值自动重写或局部重写。

7. 核心用户流程

7.1 创建作品流程

用户输入：

选择类型：玄幻 / 都市 / 仙侠 / 科幻 / 悬疑 / 女频 / 无限流等。
一句话创意，或ai生成创业。
目标字数。
目标读者。
想要的风格。
是否提供样章。
是否提供参考设定。

系统输出：

作品 DNA。
主角设定。
核心矛盾。
世界观基础规则。
全书主线。
第一卷卷纲。
前 30 章章节规划。
前 5 章详细场景卡。
前 1 章正文草稿。

用户可以修改任意部分，确认后写入正史库。

⸻

7.2 生成下一章流程

用户点击“生成下一章”。

系统自动执行：

第一步：读取当前作品状态。
包括作品 DNA、当前卷目标、当前篇章目标、最近 5 章摘要、活跃人物、未回收伏笔、读者承诺、世界规则、风格指纹。

第二步：判断下一章功能。
比如主线推进、人物冲突、伏笔提醒、伏笔回收、爽点爆发、情绪沉淀、世界扩张。

第三步：生成章节目标。
明确这一章必须发生什么、不能发生什么、结尾要给什么钩子。

第四步：生成场景卡。
每章 3 到 7 个场景，每个场景包含地点、人物、冲突、信息变化、情绪目标。

第五步：生成正文初稿。

第六步：风格改写。

第七步：审稿器检查。

第八步：不通过则自动重写。

第九步：通过后生成章节摘要、状态变化、伏笔变化。

第十步：用户确认后写入正史。

⸻

7.3 修改章节流程

用户可以对任意章节执行：

重写本章。
只改对白。
增强冲突。
增强暧昧。
增强爽点。
压缩说明。
增加细节。
改成更冷峻。
改成更有网感。
改结尾钩子。
保持剧情不变，只改文风。
保持文风不变，只改情节。

关键要求：

修改后必须重新运行状态抽取。
如果修改影响后续章节，系统必须提示受影响范围。
如果修改破坏伏笔或设定，系统必须给出风险提示。

8. 关键功能需求

作品 DNA 是最高优先级设定，所有生成必须服从。

字段：

作品名称。
类型。
子类型。
目标平台。
目标字数。
更新节奏。
核心卖点。
主角母题。
终局情绪。
主线问题。
世界观关键词。
爽点机制。
情绪机制。
禁忌规则。
风格方向。
目标读者画像。

示例：

{
  "genre": "东方玄幻",
  "core_hook": "被废的宗门杂役发现自己能听见神像遗言",

  ]
}

8.2 章节功能系统

每章必须有一个主功能和若干副功能。

主功能类型：

主线推进。
人物转折。
伏笔埋设。
伏笔回收。
爽点爆发。
危机升级。
世界观扩张。
关系推进。
反派压迫。
情绪沉淀。
阶段收束。
新篇开启。

系统需要根据当前作品状态自动推荐下一章功能。

推荐逻辑：

主线多章未推进，提高主线推进权重。
伏笔长期未出现，提高伏笔提醒权重。
连续爽点过密，提高代价和危机权重。
连续压抑过多，提高释放和阶段胜利权重。
重要角色长期未出场，提高角色回归权重。
世界观扩张过快，提高人物情绪和行动权重。

8.3 正史记忆库

正史记忆库是核心壁垒。

需要保存：

人物。
关系。
组织。
地点。
道具。
能力。
世界规则。
事件。
时间线。
伏笔。
读者承诺。
章节摘要。

关键要求：

每条正史都要有来源章节。
每条正史都要有可信状态：已确认 / 草稿 / 废弃 / 冲突。
每条正史都要支持版本变更。
每次章节修改后，需要重新计算影响。

8.5 人物一致性系统

每个重要人物都需要有“行为约束”。

字段：

目标。
恐惧。
秘密。
底线。
创伤。
诱惑。
当前信息边界。
语言习惯。
决策偏好。
关系状态。
成长阶段。

生成章节前，系统要判断角色在当前场景中会怎么做。

审稿时检查：

角色是否做了违背底线的事。
角色是否知道了不该知道的信息。
角色情绪是否跳变。
角色关系变化是否缺少过程。
角色能力是否突然增强。
角色语言风格是否漂移。

示例提示：

“风险：林昭在第 18 章明确表示不会主动求助宗门，但第 24 章直接向宗门长老求援，缺少心理转折。建议补充一个迫使他破戒的代价事件。”

8.6 伏笔账本

伏笔账本是产品的高价值模块。

每个伏笔包含：

表面意义。
真实意义。

关联事件。
预计回收区间。
当前状态。
热度分。
紧急度分。
是否提醒过。
是否已回收。

系统能力：

自动识别新增伏笔。
自动提醒长期未出现伏笔。
自动建议伏笔回收窗口。
自动判断伏笔是否过早暴露。
自动生成伏笔加深桥段。
自动生成误导性解释。
自动生成阶段性回收方案。

伏笔状态：

未激活。
已埋设。
已提醒。
已加深。
部分回收。
完整回收。
废弃。
冲突。

8.7 风格指纹

用户可以通过两种方式创建风格：

第一，选择系统预设风格。
例如冷峻升级流、轻松吐槽流、古典仙侠感、黑暗悬疑感、都市爽文感、细腻情绪流。

第二，上传样章提取风格。
系统抽取叙述距离、句长、对白比例、情绪强度、修辞偏好、节奏方式、章末钩子方式。

风格指纹字段：

叙述视角。
叙述距离。
句长分布。
对白比例。
心理描写比例。
动作描写比例。
环境描写比例。
信息密度。
情绪外露程度。
幽默程度。
修辞系统。
常用词。
禁用词。
章末风格。
战斗描写风格。
暧昧描写风格。
悬疑揭示风格。

每章生成后计算 Style Drift Score。
如果偏移过高，系统自动进入风格重写。

8.8 审稿器

审稿器由 Auditor 执行，但内部是多个工具检查项，并非多个 Agent。

检查维度：

设定一致性。
人物一致性。
时间线一致性。
地点一致性。
道具体系一致性。
能力体系一致性。
伏笔状态。
主线推进。
人物变化。
冲突强度。
爽点密度。
节奏风险。
重复桥段。
风格偏移。
水文风险。
章末钩子。
下一章承接。

审稿结果分为：

绿色：可发布。
黄色：建议修改。
红色：必须重写。

    {

8.9 章节质量评分

每章生成后打分。

建议公式：

Chapter Quality Score =
0.15 * 主线推进
0.15 * 人物变化
0.12 * 冲突强度
0.12 * 章末钩子
0.10 * 风格一致性
0.10 * 设定一致性
0.08 * 信息增量
0.08 * 情绪张力
0.05 * 新鲜度

阈值：

90 分以上：强推荐发布。
80 到 89 分：可发布，建议微调。
70 到 79 分：需要修改。
70 分以下：自动重写。

注意：评分重点不是文学奖标准，而是连载任务完成度。

8.10 套路变体生成器

系统要识别桥段类型，避免重复。

常见桥段：

拍卖会争夺。
秘境试炼。
宗门大比。
身份揭露。

强者救场。
反派压迫。
女主误会。
交易谈判。
逃亡追杀。
闭关突破。

每次使用桥段后，系统记录使用次数和最近使用章节。

当桥段重复时，系统自动生成变体。

示例：

原桥段：主角被当众嘲讽，然后展示实力打脸。
变体一：主角故意隐藏实力，反向诱敌。
变体二：配角替主角出手，主角在暗处布局。
变体三：主角赢了测试，但暴露了更危险的秘密。
变体四：反派没有震惊，说明他早已知道主角底牌。
变体五：主角选择不赢，因为真正目标是破坏测试规则。

18. 风险与解决方案

最大风险一：生成质量不稳定。
解决方案：不要依赖一次生成。必须使用“规划 → 生成 → 审稿 → 重写 → 入库”的流程。

最大风险二：上下文越来越大，成本越来越高。
解决方案：结构化正史 + 精准召回。每次只喂相关人物、相关伏笔、相关规则，而不是整本书。

最大风险三：用户觉得系统太复杂。
解决方案：前台简化成“生成下一章”“审稿”“修改”“确认入库”；后台复杂度隐藏。

最大风险四：AI 文风同质化。
解决方案：风格指纹 + 风格偏移检测 + 样章风格提取。

最大风险五：多 Agent 拖累。
解决方案：采用三角色模式，不做多 Agent 群聊。所有细分能力工具化。

最大风险六：长篇后期崩盘。
解决方案：伏笔账本、承诺账本、人物弧线、节奏曲线、世界规则检查必须在 V1 就做基础版。

20. 交互设计核心

用户不应该频繁写 prompt。

错误交互：

用户：帮我写下一章。
AI：好的，请告诉我人物、背景、风格、情节。

正确交互：

用户点击：生成下一章。
系统自动读取所有上下文。
系统给出章节目标、正文、审稿报告。
用户只需要选择：接受、重写、局部修改、改方向。

产品要做到：

用户负责决策，系统负责执行。

并且一切都是由云服务提供商提供的大模型api驱动的，要做好兼容openai的api接入，然后用ai大模型驱动所有的功能执行。

22. 目前这个 PRD 的关键产品主张

我建议你们把核心卖点定成这句话：

“不是帮你写一章，而是帮你养活一部长篇。”

更商业化一点：

“面向百万字连载的 AI 小说生产系统：自动写作、自动记忆、自动审稿、自动维护。”

更有冲击力一点：

“让 AI 持续写 300 章也不崩。”

内部产品原则：

生成只是入口。
记忆才是中枢。
维护才是壁垒。
审稿才是质量闸门。
风格才是作品辨识度。
节奏才是商业生命线。
